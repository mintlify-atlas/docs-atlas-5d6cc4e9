## Getting Started

### Prerequisites

Before contributing code, ensure you have:

<Steps>
  <Step title="Development Environment">
    - **Node.js 18** or higher
    - **Git** for version control
    - **Code editor** (VS Code recommended)
    - **Docker** and **Docker Compose** installed
    - **NVIDIA GPU** with proper drivers for testing
  </Step>
  
  <Step title="GitHub Account">
    - Create a [GitHub account](https://github.com) if you don't have one
    - Set up [SSH keys](https://docs.github.com/en/authentication/connecting-to-github-with-ssh) (recommended)
    - Configure Git with your name and email:
    
    ```bash
    git config --global user.name "Your Name"
    git config --global user.email "your.email@example.com"
    ```
  </Step>
  
  <Step title="Fork the Repository">
    1. Visit [github.com/duixcom/Duix.Avatar](https://github.com/duixcom/Duix.Avatar)
    2. Click the "Fork" button in the top-right corner
    3. Clone your fork locally:
    
    ```bash
    git clone https://github.com/YOUR_USERNAME/Duix.Avatar.git
    cd Duix.Avatar
    ```
  </Step>
  
  <Step title="Set Up Development">
    ```bash
    # Install dependencies
    npm install
    
    # Set up Docker services
    cd deploy
    docker-compose up -d
    
    # Return to project root
    cd ..
    
    # Start development
    npm run dev
    ```
  </Step>
</Steps>

## Development Workflow

### 1. Create a Branch

Always create a new branch for your work:

```bash
# Update your fork
git checkout main
git pull origin main

# Create a new branch
git checkout -b feature/your-feature-name
# or
git checkout -b fix/issue-number-description
```

**Branch naming conventions:**
- `feature/` - New features
- `fix/` - Bug fixes
- `docs/` - Documentation updates
- `refactor/` - Code refactoring
- `test/` - Test additions or updates

### 2. Make Your Changes

<AccordionGroup>
  <Accordion title="Code Guidelines">
    - Follow existing code style and formatting
    - Write clear, descriptive variable and function names
    - Add comments for complex logic
    - Keep functions small and focused
    - Avoid unnecessary dependencies
  </Accordion>

  <Accordion title="Testing Your Changes">
    Before submitting:
    
    1. **Build the application:**
    ```bash
    npm run build
    ```
    
    2. **Test locally:**
    - Test the feature/fix works as expected
    - Verify no new bugs are introduced
    - Test on both Windows and Linux if possible
    
    3. **Check for errors:**
    - Review client logs
    - Check Docker service logs
    - Test with different GPU configurations if applicable
  </Accordion>

  <Accordion title="Documentation Updates">
    If your change affects user-facing functionality:
    
    - Update relevant README sections
    - Add/update code comments
    - Document new features or API changes
    - Update configuration examples if needed
  </Accordion>
</AccordionGroup>

### 3. Commit Your Changes

```bash
# Stage your changes
git add .

# Commit with a clear message
git commit -m "feat: add support for custom avatar backgrounds"
# or
git commit -m "fix: resolve issue #123 - Docker service connection error"
```

**Commit message format:**
- `feat:` - New feature
- `fix:` - Bug fix
- `docs:` - Documentation changes
- `style:` - Code style/formatting
- `refactor:` - Code refactoring
- `test:` - Test updates
- `chore:` - Maintenance tasks

### 4. Push and Create Pull Request

```bash
# Push to your fork
git push origin feature/your-feature-name
```

Then:

1. Go to [github.com/duixcom/Duix.Avatar](https://github.com/duixcom/Duix.Avatar)
2. Click "Pull requests" → "New pull request"
3. Click "compare across forks"
4. Select your fork and branch
5. Fill in the PR template:
   - Clear description of changes
   - Related issue numbers
   - Testing performed
   - Screenshots/videos if applicable

<Info>
  Pull requests are reviewed by maintainers. Be patient and responsive to feedback.
</Info>

## Reporting Issues

### Before Reporting

<Steps>
  <Step title="Search Existing Issues">
    Check if the issue already exists:
    - [Open Issues](https://github.com/duixcom/Duix.Avatar/issues?q=is%3Aissue+is%3Aopen)
    - [Closed Issues](https://github.com/duixcom/Duix.Avatar/issues?q=is%3Aissue+is%3Aclosed)
  </Step>
  
  <Step title="Complete Self-Check">
    Follow the [FAQ self-check steps](/resources/faq#self-check-steps-before-asking-questions):
    - Verify Docker services are running
    - Confirm GPU and drivers are working
    - Update to latest version
  </Step>
  
  <Step title="Gather Information">
    Collect:
    - Operating system and version
    - GPU model and driver version
    - Duix Avatar version
    - Docker service status
    - Error logs (client and server)
    - Steps to reproduce
  </Step>
</Steps>

## Technical Architecture

### Project Structure

```
Duix.Avatar/
├── src/
│   ├── main/
│   │   ├── service/
│   │   │   ├── model.js      # Model training APIs
│   │   │   ├── video.js      # Video synthesis APIs
│   │   │   └── voice.js      # Voice/TTS APIs
│   │   └── ...
│   ├── renderer/             # Client UI
│   └── ...
├── deploy/
│   ├── docker-compose.yml        # Full version
│   ├── docker-compose-lite.yml   # Lite version
│   └── docker-compose-linux.yml  # Linux version
├── README.md
└── package.json
```

### Key Components

<Tabs>
  <Tab title="Client Application">
    Built with Electron and Node.js:
    - Desktop application (Windows/Linux)
    - User interface for avatar creation
    - Video project management
    - Communicates with Docker services
  </Tab>
  
  <Tab title="Docker Services">
    Three microservices:
    
    1. **Duix.Avatar-asr** (ASR - Automatic Speech Recognition)
       - Port: Internal
       - Based on fun-asr
       - Converts speech to text
    
    2. **Duix.Avatar-tts** (TTS - Text-to-Speech)
       - Port: 18180
       - Based on fish-speech-ziming
       - Voice cloning and synthesis
    
    3. **Duix.Avatar-gen-video** (Video Generation)
       - Port: 8383
       - Core avatar video synthesis
       - Lip-sync and video processing
  </Tab>
  
  <Tab title="APIs">
    RESTful APIs exposed by Docker services:
    
    - Model training API
    - Audio synthesis API
    - Video synthesis API
    - Progress query API
    
    See [Open APIs documentation](https://github.com/duixcom/Duix.Avatar#open-apis)
  </Tab>
</Tabs>

### Technology Stack

- **Frontend:** Electron, HTML/CSS/JavaScript
- **Backend:** Node.js
- **AI Services:** Python, PyTorch, CUDA
- **Containerization:** Docker, Docker Compose
- **ASR:** fun-asr
- **TTS:** fish-speech-ziming

## Development Tips

<AccordionGroup>
  <Accordion title="Working with Docker Services">
    **View service logs:**
    ```bash
    docker logs duix-avatar-asr -f
    docker logs duix-avatar-tts -f
    docker logs duix-avatar-gen-video -f
    ```
    
    **Restart a service:**
    ```bash
    docker-compose restart duix-avatar-asr
    ```
    
    **Rebuild services:**
    ```bash
    docker-compose down
    docker-compose up -d --build
    ```
  </Accordion>

  <Accordion title="Client Development">
    **Run in development mode:**
    ```bash
    npm run dev
    ```
    
    **Build for production:**
    ```bash
    npm run build
    ```
    
    **Access client logs:**
    Check the application's log directory (varies by OS)
  </Accordion>

  <Accordion title="Testing API Endpoints">
    Use curl or Postman to test APIs:
    
    ```bash
    # Test TTS service
    curl -X POST http://127.0.0.1:18180/v1/invoke \
      -H "Content-Type: application/json" \
      -d '{...}'
    
    # Test video generation
    curl -X POST http://127.0.0.1:8383/easy/submit \
      -H "Content-Type: application/json" \
      -d '{...}'
    ```
    
    See [Open APIs](https://github.com/duixcom/Duix.Avatar#open-apis) for full API documentation.
  </Accordion>

  <Accordion title="Debugging Tips">
    - Use browser DevTools for client debugging (Ctrl+Shift+I)
    - Check Docker service logs for backend issues
    - Monitor GPU usage with `nvidia-smi`
    - Test with different hardware configurations
    - Use verbose logging when available
  </Accordion>
</AccordionGroup>
